Description

This project was something I came up with to put into practice what I’d learned about the back and the front end in an idea of my own. The app is functional now although it still needs further tweaks, improvements and features that I’ll mention below. Throughout making this web application I learned a huge amount, especially my familiarity with JS (the front end uses vanilla JS and the back end uses Node.js) and improving my understanding of how various web technologies work in practice.

Early on in the project I learned hands-on about how to improve my database schema design. My first effort used too many levels of nesting in embedded objects and was inefficient when queried. I remade my early version of the app with a new schema design involving less embedding, replaced with more linked collections and some denormalization, that allowed for more efficient processing on the back-end.

I spent some time optimising the questions part of the site by having the data for newly answered questions of a user be kept locally on the client and shared between the different questions modes on the page. The user’s answers data is then only POSTed to the server and subsequently saved in the database every set period of time, or when the page is navigated away from, to reduce the number of requests to the server. On the back-end, the server keeps track of users currently in a session answering questions, and for which category, to reduce the number of queries on the database.

The app uses multiple third-party APIs to provide the questions for most of the categories, although the app is also flexible in allowing categories to use questions I manually provide in the database (or a mix of both; eg. database for the suggestions mode and then an API for the search mode, such is the case with the Books category). The app also uses Google OAuth 2.0, the Google places API (for retrieving user locations at registration) and Passport for user sessions.

Once I was largely happy with this project I did some work on optimising the front end using Google Lighthouse. All pages now score 90+ for each Lighthouse metric (nearly all pages score 100 for all metrics except performance). In addition to the performance optimisations I’ve already made, such as: bundling JS, asynchronously loading certain assets, and adjusting my code to only lazy-load images that don’t form part of the first contentful paint of a page, among other things - I know that there are further performance improvements I can still make, such as inlining the critical CSS of each page.

Although the app functions, there are features and improvements I plan to work on (some of which are essential and I simply haven’t gotten round to yet), such as:

  • Account features (allowing user to change their password and other account details) as well as account email verification on signup.
  • Sanitising user input on the relevant pages.
  • Optimising the back end for the Find Kindred and Recommendations routes.
  • Further front end optimisation.

Please note that if you do test the app, the recommendations etc. that it generates for you will not be of any real relevance, as most of the data currently in the app is for bot accounts with randomly generated answer data.
